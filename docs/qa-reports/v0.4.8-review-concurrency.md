# v0.4.8 동시성 안전성 리뷰

## 요약

**2개 이슈 발견**: (1) TaskGroup 내부 비동기 Resolver 호출의 잠재적 경합 위험 (MED), (2) Control chat relay 클로저의 actor context 명확성 부족 (LOW).

---

## 발견 사항

### 1. **dispatchRelaysInParallel 의 TaskGroup 전외 resolver 호출 (MED)**

**위치**: `DispatchService.swift:150-155`

```swift
for relay in relays {
    guard (try? await resolver.resolve(agent: relay.target)) != nil else {
        await observer.relaySkipped(...)
        continue
    }
    // ↓ group.addTask 외부에서 수행
}
```

**문제**:

- `resolver.resolve(agent:)` 는 TaskGroup **외부**에서 `await` 됨.
- TaskGroup 생성 전 직렬 실행되므로 (의도된 것으로 보임), 자체로는 경합이 아니나, **코드 의도**가 명확하지 않음.
- 주석 부재로 인해 "왜 loop 밖인가"를 미래 메인테이너가 오해할 위험.

**개선**:

```swift
// v0.4.8: 릴레이 대상 사전검증 — group 시작 전 unknown agent skip
// (resolver.resolve 호출이 비동기이지만, TaskGroup 내부에서는
// 각 relayInternal 호출만 병렬화).
for relay in relays {
    guard (try? await resolver.resolve(...)) != nil else { ... }
    // addTask 로직
}
```

### 2. **wireControlMainChatRelay 의 클로저 actor context (LOW)**

**위치**: `ControlTowerEnvironment+Dispatch.swift:54-79`

```swift
chatVM.onAssistantResponseComplete = { [weak service, weak chatVM, weak folderViewModel] body in
    // 여기서 실행 컨텍스트는? @MainActor인가?
    let parsed = parser.parse(body)
    await withTaskGroup(...) { group in
        // group.addTask { [service] in ... }
    }
}
```

**문제**:

- 클로저 자체에 `@MainActor` 또는 `@Sendable` 표시 없음 → 호출 컨텍스트에 의존.
- `ChatViewModel.onAssistantResponseComplete` 가 @MainActor 컨텍스트인지, 일반 actor인지 명시되지 않음.
- `chatVM.appendRelayResult` 호출이 @MainActor 메서드라면, 클로저도 @MainActor 여야 함.

**개선**:

- `ChatViewModel.onAssistantResponseComplete` 의 타입 선언에 명시적 actor isolation 추가.
- 또는 클로저 내부 `chatVM.appendRelayResult` 호출을 `await MainActor.run { ... }` 로 감싸기.

---

## 안전한 부분

✅ **DispatchService 자체는 actor-isolated** — `dispatchInternal` 에 대한 동시 호출은 actor mailbox 로 직렬화됨. 각 relay 의 `dispatchInternal` 호출이 TaskGroup 내부에서 병렬화되지만, 호출 대상이 서로 다른 agent (다른 process) 이므로 race 없음.

✅ **TaskGroup 결과 수집** — `for await result in group` 는 모든 작업 완료를 기다리므로, 부분적 결과로 인한 관찰 순서 비결정성이 있어도 `relayResultArrived` 호출이 serialized 되므로 observer 상태 관리는 안전.

✅ **Relay depth cap** — 4단계 제한으로 exponential dispatch 폭발 방지.

---

## 권장 개선

1. **dispatchRelaysInParallel 라인 150**: resolver 호출을 `// v0.4.8: ` 주석으로 명시.
2. **ChatViewModel.onAssistantResponseComplete**: @MainActor 또는 명시적 actor isolation 애노테이션 추가.
3. 향후: TaskGroup 내 resolver 호출이 필요해지면, `group.addTask { [relay] in (try? await resolver.resolve(agent: relay.target)) != nil ? ... }` 로 리팩터링.
