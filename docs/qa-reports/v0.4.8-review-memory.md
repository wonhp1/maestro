# v0.4.8 메모리/라이프사이클 리뷰

## 발견된 이슈

### 1. **DispatchService.dispatchRelaysInParallel — weak self 안전성** (LOW)

- **L164**: `group.addTask { [weak self] in ... }`에서 `guard let self` 후 사용 ✓ 안전
- `withTaskGroup` 스코프 내에서 task 모두 완료될 때까지 group 유지 → self 참조 유효
- 릴레이 깊이 cap(기본 4)이 있어 runaway 없음 ✓

### 2. **ControlTowerEnvironment+Dispatch.wireControlMainChatRelay — retain cycle 위험** (MEDIUM)

**문제:**

- **L54**: `chatVM.onAssistantResponseComplete = { [weak service, weak chatVM, weak folderViewModel] body in ... }`
  - Outer 클로저가 weak 캡처 ✓ 안전
  - **BUT L60**: `group.addTask { [service] in ... }` — service를 **strong 캡처**
  - group.addTask 클로저가 group 완료될 때까지 service 유지 (의도한 설계)

**라이프사이클 위험:**

- `onAssistantResponseComplete` 클로저가 `chatVM` 내부에 저장되고, control 폴더 evict 시 chatVM dealloc
- 만약 `ChatViewModel` 내부에서 `self.onAssistantResponseComplete = nil` cleanup이 **없다면** 클로저 메모리 누수 가능
- wireControlMainChatRelay가 bootstrap에서만 호출되는지 확인 필요 (매번 wire/re-wire되면 중복 설정)

**권장:**

1. `ChatViewModel`에서 deinit 또는 cleanup 메서드로 `self.onAssistantResponseComplete = nil` 명시 필요
2. wireControlMainChatRelay가 한 번만 호출됨을 보장하는 guard 추가 (예: flag 또는 Optional check)

### 3. **group.addTask [service] strong capture의 의도 검증** (LOW)

- **L60, L62**: task 그룹 내에서 service를 strong으로 잡는 것은 relay 완료 때까지 service 살아있게 함 (설계상 OK)
- `try? await service.dispatch(...)` 이후 task 완료 → service 해제
- 문제 없으나, intent 주석 추가하면 가독성↑

### 4. **AsyncTaskGroup 컨텍스트 이탈** (LOW)

- **L71-76**: `for await (target, reply) in group` 루프 내에서 `chatVM.appendRelayResult` 호출
- chatVM이 weak 참조라 이 사이에 dealloc 가능 → 추가 guard 필요 (현재 없음)

**권장:**

```swift
for await (target, reply) in group {
    guard let reply, let chatVM else { continue }  // chatVM 재확인
    chatVM.appendRelayResult(from: label, body: reply.body)
}
```

## 요약 (300단어 이하)

**메모리:** DispatchService의 relay 병렬화는 weak self 처리가 안전함. 하지만 control chat 클로저 wiring에서 라이프사이클 취약점 2개 발견.

1. **ChatViewModel의 onAssistantResponseComplete 클로저 cleanup 부재** — evict 시 메모리 누수 가능 (MEDIUM)
2. **group 루프 중 chatVM weak 참조 재확인 없음** — dealloc 후 appendRelayResult 호출 미방지 (LOW)

**권장 개선:**

- ChatViewModel.deinit에서 `self.onAssistantResponseComplete = nil` 명시
- wireControlMainChatRelay 내 bootstrap guard 추가 (중복 wire 방지)
- L71 루프에서 `guard let chatVM else { continue }` 추가

현재 코드는 정상 케이스에서 동작하지만, control 폴더 리셋 또는 빠른 evict 시나리오에서 dangling 참조 위험 있음.
